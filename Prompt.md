# Prompt

## 目标

- 为 `gpt-image-2` 建立按分辨率区分的路由与收费策略，而不是继续沿用“同模型统一处理、统一定价”。

## 已确认需求

- `1k`：走免费账号渠道。
- `2k`、`4k`：走 `Codex` 和 `APIMart` 渠道。
- `2k`、`4k` 若外置渠道失败，允许继续回落 `free runner`。
- 下游面向用户的默认收费调整为后台额度价（⭐）：
  - `1k`：默认 `0.06` ⭐
  - `2k`：默认 `0.10` ⭐
  - `4k`：默认 `0.20` ⭐
- 后台可通过 `model_price` 覆盖分档价格，建议键名使用 `gpt-image-2:1k`、`gpt-image-2:2k`、`gpt-image-2:4k`；例如 `gpt-image-2:1k=0` 表示 1K 免费。

## 涉及范围

- 当前号池 `gpt2api`：
  - `gpt-image-2` 路由选择
  - 分辨率到渠道策略映射
  - 号池侧成本/计费语义是否需要随分辨率变化
- 下游后端 `new-api`：
  - 用户扣费规则
  - 向前端暴露的价格/能力说明
- 下游前端 `new-api-web`：
  - 分辨率收费提示
  - 渠道能力或失败提示文案

## 当前边界

- 当前号池代码里的图片计费是按 `model.image_price_per_call * n` 统一计算，尚不是天然的“同模型按 resolution 分档计费”。
- 当前变更不是单点改价，而是“分辨率路由 + 分辨率收费 + 前端提示”的跨系统联动。
- 下游后端 `new-api` 已是预扣费模型，因此用户侧收费判断必须考虑“创建任务时如何预扣、任务结束时如何结算/退款”，不能只看号池最终命中了哪条渠道。
- 下游旧配置或历史 smoke 中出现的 `gpt-image-2 model_price=0.12` 只代表旧单价状态；本轮分档默认值以 `1k=0.06`、`2k=0.10`、`4k=0.20` ⭐ 为准，但最终价格以后台 `model_price` 分档配置为真相源。
- ChatGPT Web Runner / free runner 兜底不能保证原生稳定 2K/4K 能力；`2k/4k` 的原生大图预期主要依赖 Codex / APIMart，fallback 是完成率兜底，不是高分辨率能力真相源。

## 计费原则

- 用户侧收费真相源放在下游后端 `new-api`，按“用户请求的 resolution 档位”预扣，而不是按号池最终命中的 provider 反推价格。
- 号池 `gpt2api` 负责路由与执行，并返回任务状态、最终命中链路、必要的分辨率/兜底信息，供下游结算核对。
- 既然已确认 `2k/4k` 允许回落 `free runner`，默认语义应视为“用户购买的是 2k/4k 档位服务”，不是“购买某个具体 provider”；因此成功时通常按请求档位结算，失败时再退款。
